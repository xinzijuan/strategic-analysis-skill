# 战略分析技能

## 适用场景

当用户需要做以下类型分析时使用此技能：

- 行业研究和市场分析
- 竞品分析和竞争格局判断
- 商业模式分析和战略选择
- 产品/技术路线决策
- 投资并购和合作策略
- 企业转型和增长策略

---

## 工作原则

1. **需求深度理解**：在开始分析前，先通过问卷确认关键信息，确保分析精准
2. **金字塔逻辑**：采用自上而下的结构化思维，结论在顶端，论据和数据在下方
3. **MECE原则**：确保论证相互独立、完全穷尽、共同作用
4. **假设驱动**：先提出假设，再通过数据验证
5. **冷静深度**：采用科技公司偏好的理性、数据驱动、深度分析风格

---

## 执行步骤

### 第一阶段：前置确认

**目的**：通过问卷确认分析需求的关键信息，确保分析精准，避免走弯路

1. **分析需求确认**
   通过问卷收集以下关键信息：

   | 确认项 | 说明 | 输出类型 |
   |---------|------|---------|
   | 话题定义 | 用户想研究的具体内容 | 文本 |
   | 边界条件 | 分析范围（行业、公司、时间、地域等） | 文本 |
   | 汇报场合 | 会议性质（内部讨论/高管汇报/客户会议/公开演讲等） | 选择 |
   | 质量要求 | 深度（初步/深入/全面）、格式要求、交付时间 | 文本 |
   | 参会人 | 谁会参与、需要了解他们的关注点 | 文本 |
   | 历史讨论 | 是否有之前类似的讨论、结论或决策 | 文本 |
   | 相关素材 | 是否有相关的数据、报告、案例、文档 | 文本 |
   | 其他要求 | 特殊需求或偏好 | 文本 |

2. **问卷内容构建**
   根据确认项生成结构化问卷

3. **确认与记录**
   - 向用户展示确认内容
   - 在分析文档中记录关键确认点

4. **如有历史信息**
   - 如果有历史讨论或相关素材，记录来源
   - 判断是否需要发出来帮助学习

### 第二阶段：需求理解与议题拆解

5. **确认核心问题**
   - 基于前置确认的信息，明确用户想解决的战略问题
   - 确认分析范围（行业、公司、时间跨度）
   - 确认分析深度（初步/深入/全面）

6. **金字塔拆解议题树**
   - 将核心问题置于顶端
   - 向下拆解为3-5个关键子问题
   - 确保子问题MECE（相互独立、完全穷尽）
   - 每个子问题向下继续拆解论据

7. **与用户确认议题树**
   - 展示金字塔结构的议题树
   - 确认是否需要调整
   - 用户确认后进入研究阶段

### 第三阶段：案头研究

8. **数据搜集（参考麦肯锡方法）**
   - 参考 `references/research-methods.md` 的研究方法
   - 搜集行业数据、市场报告、竞品信息
   - 记录数据来源和可信度

9. **交叉验证**
   - 多源验证关键数据
   - 识别数据之间的矛盾点
   - 在 `progress.md` 中记录验证过程

10. **形成初步假设**
    - 基于数据形成各子问题的初步假设
    - 识别假设的不确定性

### 第四阶段：深度分析与假设验证

11. **Case Study式分析**
    - 对每个判断进行Case Study式验证
    - 提出假设 → 搜集数据 → 验证假设 → 得出结论
    - 使用 `references/analysis-frameworks.md` 的分析框架
    - 参考 `references/mckinsey-methods.md` 的麦肯锡方法论

12. **MECE论证**
    - 确保论据相互独立（Mutually Exclusive）
    - 确保论据完全穷尽（Collectively Exhaustive）
    - 确保论据共同作用（Collectively）
    - 形成严谨的逻辑链

### 第五阶段：输出storyline

13. **构建金字塔式storyline**
    - 参考 `assets/storyline-template.md` 的格式
    - 顶部是核心判断
    - 向下是支撑论据
    - 最底层是数据和来源

14. **评审storyline**
    - 检查MECE原则是否满足
    - 检查假设是否被验证
    - 检查论证是否完整

15. **与用户确认核心判断**
    - 展示金字塔结构的storyline
    - 确认是否需要调整
    - 用户确认后进入文档生成阶段

### 第六阶段：生成Word文档

16. **生成Word文档**
    - 参考 `assets/word-template.md` 的格式
    - **输出完整的研究内容**，包括详细论证、数据来源、图表说明
    - 采用金字塔结构：顶部核心判断→中部论证→底部数据
    - 包含详细的论证逻辑链
    - 每个数据标注来源
    - 使用烟蓝色配色方案
    - **生成.docx文件保存到桌面**
    - 文件名格式：[主题]战略分析.docx

17. **输出评审**
    - 检查Word文档内容是否完整
    - 检查数据是否准确、来源是否清晰
    - 检查MECE是否满足
    - 检查是否有重复内容

### 第七阶段：后续确认

**目的**：向用户确认研究结论和假设的正确性，用于后续应用或学习

18. **展示核心判断**
    - 向用户展示核心判断和主要论据
    - 确认关键数据、逻辑链是否合理

19. **验证假设准确性**
    - 询问用户研究结论是否正确
    - 确认用户的初始假设是否被验证

20. **回应修正意见**
    - 回答用户提出的任何疑问
    - 根据用户反馈调整内容
    - 将反馈记录用于后续改进

21. **输出终稿**
    - 根据确认输出最终版本Word文档

### Storyline格式（金字塔结构）

按照 `assets/storyline-template.md` 格式输出，遵循金字塔原理：

- **顶层（核心判断）**：1-2个核心观点，简洁有力
- **中部（关键论据）**：3-5个MECE的支撑论据
- **底层（数据支撑）**：具体数据点，带来源

### Word文档格式（详细论证）

按照 `assets/word-template.md` 格式输出：

| 页码 | 标题 | 核心内容 |
|-----|------|---------|
| 封面 | 标题+副标题+日期+汇报人 | - |
| 1 | 核心判断 | 1-2个核心观点 + 数据概览表格 |
| 2 | 市场分析-市场规模 | TAM/SAM/SOM + 增长趋势分析 |
| 3 | 市场分析-竞争格局 | 竞对对比表格 + 机会点分析 |
| 4 | 核心判断1 | 完整论证逻辑链 + 论据表格 |
| 5 | 核心判断2 | 关键洞察 + 反例分析 + 边界条件 |
| 6 | 风险分析 | 风险矩阵表格 + 应对建议 |
| 7 | 战略建议 | 3个建议（优先级排序+详细理由） |
| 8 | 行动路线图 | 阶段规划表格 + 预期成果 |
| 附录1 | 数据来源汇总 | 数据来源表格 |
| 附录2 | 后续研究 | 待研究问题列表 |

**Word文档核心要求**：
- **内容全面性**：覆盖所有关键维度，不遗漏重要论据
- **逻辑合理性**：论证逻辑链清晰，因果关系明确，MECE原则
- **观点独到性**：提供独特洞察，避免泛泛而谈，给出明确判断
- **数据支撑**：每个关键判断有具体数据支撑，标注来源
- **论证深度**：包含完整论证过程，不是结论堆砌
- **反例分析**：识别边界条件和例外情况，增强论证严谨性

**Word文档简化要求**：
- **图表简化**：可使用简单表格呈现数据，不需要复杂图表
- **重点在内容**：图表服务于观点表达，不追求视觉效果
- **表格为主**：优先使用表格呈现对比数据、论据、风险等
- **逻辑优先**：强调论证过程和逻辑链，而非数据可视化

**输出要求**：
- 输出.docx文件保存到桌面
- 文件名格式：[主题]战略分析.docx
- 使用统一字体和段落格式
- 不使用任何emoji符号

**风格要求**：
- **金字塔结构**：顶部观点→中部论证→底部数据
- **MECE原则**：论据相互独立、完全穷尽、共同作用
- **假设驱动**：先假设后验证
- **理性、数据驱动**：每个判断有数据和逻辑支撑
- **每数据必有来源**：数据服务于观点，带真实来源
- **每关键点必有highlight**：使用TC Red色块/粗体/边框突出
- **完整论证**：避免大段文字，使用要点和表格
- **不使用任何emoji符号**
- **专业排版**：使用统一的字体、颜色、间距

## 文档设计规范

### 视觉设计
- **配色方案**：烟蓝色系（不同饱和度和深浅）
  - 主色：烟蓝 #6B8EAE（中饱和度）
  - 深色：烟蓝深 #4A7698（深色）
  - 浅色：烟蓝浅 #C3D9E8（浅色）
  - 背景色：白色 #FFFFFF
  - 辅助色：灰色 #808080
- **Highlight色（TC Red）**：#E31E24
- **字体层级**：一级标题18pt+，二级标题16pt，正文12pt，注释10pt
- **留白原则**：页面内容适度，避免拥挤

### Highlight机制
- 核心观点使用TC Red粗体
- 关键数据使用TC Red色块背景
- 重要结论使用TC Red边框突出
- 每页Highlight不超过2个重点

## MECE原则应用

在论证过程中必须遵循MECE原则：

| 原则 | 说明 | 应用方法 |
|-----|------|---------|
| **M** - Mutually Exclusive | 相互独立 | 检查论据之间是否有重叠 |
| **E** - Collectively Exhaustive | 完全穷尽 | 检查是否遗漏重要论据 |
| **C** - Collectively | 共同作用 | 所有论据共同支撑核心观点 |
| **E** - Exhaustively | 穷尽探索 | 确保所有角度都考虑 |

## 能力边界

此技能不适合以下场景：

- 需要实时交易数据的分析
- 需要访问私有数据（除非用户提供）
- 需要预测未来的具体数值
- 需要法律、财务等专业领域的专业判断

## 质量标准

输出需要满足以下标准（参考 `references/rubric.md`）：

- D1: 核心判断清晰且有洞察
- D2: 论据充分且来源可靠
- D3: 逻辑自洽且结构清晰
- D4: 数据准确且有验证
- D5: 观点独特且有价值
- D6: 表达精准且符合风格
- D7: 无重大事实错误
- D8: Word文档内容完整，无重复内容
- D9: 不使用任何emoji符号
- D10: 使用烟蓝色配色方案
- D11: 图表必须有具体数据（如使用表格）
- **D12: 符合MECE原则**
- **D13: 采用金字塔结构**
- **D14: 假设驱动，验证闭环**
- **D15: 完成前置需求确认**
- **D16: 内容全面、逻辑合理、观点独到**
- **D17: 每数据必有来源**

---

## 异常处理

- 如果关键数据无法获取，说明情况并使用替代数据或假设
- 如果研究过程中发现议题树需要调整，及时与用户确认
- 如果发现数据矛盾，在 `progress.md` 中记录并说明处理方式
- 如果MECE不满足，重新拆解论据
- 如果前置确认发现重要信息，及时调整分析方向

## 工作流程

1. **前置需求确认** → 展示问卷 → 收集确认信息 → 记录关键点
2. 确认核心问题 → 3. 金字塔拆解议题树（MECE） → 4. 用户确认
3. 案头研究（含数据搜集） → 5. 提出假设 → 6. 验证假设（Case Study） → 7. 输出金字塔storyline
4. 用户确认核心判断 → 8. 生成Word文档 → 9. 评审输出 → 10. 输出终稿
